# P7.2 — Hetzner threshold alerts + PM2 log rotation

**Task:** #345 — CPU > 80% / RAM > 90% / disk > 85% / swap > 50% alerts into Telegram via `@Jarvis_bectly_bot`, plus log rotation for PM2 (`/var/log/pm2/`), app cron (`/var/log/janicka-shop/`), and nginx (`/var/log/nginx/janicka-shop.*.log`) — all 30d retention, 10 MB/file trigger, gzip on rotate.
**Acceptance:** `df -h` shows log dirs stable in size after a synthetic rotation (`logrotate -f`) and post-stress-test run; `stress-ng --cpu $(nproc) --timeout 600s` drives 1-min load over the CPU threshold and triggers a Telegram alert within ≤ 15 min.
**Status:** READY-TO-EXECUTE on VPS (46.224.219.3). No domain dependency — all probes use direct IP / localhost. Pairs with P7.1 (which owns up/down + SSL alerts); P7.2 is purely the system-health + log-hygiene layer.

---

## Decision: which threshold stack

Pick ONE of the two paths. Option A is the free-tier, zero-infra default and satisfies acceptance. Option B is the deferred "proper" upgrade — do it later if the single-metric keyword-monitor becomes too coarse.

### Option A — UptimeRobot keyword-threshold on `/api/health-extended` (recommended)

- **Why:** reuses the P7.1 Telegram alert contact, no new daemon on the box, ships today. Single probe covers CPU / RAM / disk.
- **How:** Next.js route `src/app/api/health-extended/route.ts` (✅ shipped this cycle) returns JSON with `cpu_load_pct`, `mem_pct`, `disk_pct`, plus `"status":"breach"` when ANY metric crosses its configured threshold. Protected by `CRON_SECRET` bearer — UptimeRobot sends the header via "Custom HTTP Headers".
- **Thresholds:** baked into the route (`THRESHOLDS = { cpuLoadPct: 80, memPct: 90, diskPct: 85, swapPct: 50 }`). Change by editing route + redeploying — no dashboard edits.
- **Alert trigger:** UptimeRobot keyword-monitor watches for `"status":"ok"` on the response. When a threshold breaches, the route flips to `"status":"breach"` → keyword stops matching → monitor goes Down → existing Telegram alert contact fires.
- **Swap:** Node's `os` module doesn't expose swap by default; we surface `/proc/meminfo` `SwapFree`+`SwapTotal` via the same route (add in follow-up if the VPS actually uses swap — current Hetzner box has swap disabled in the base image, so deferred).

### Option B — node_exporter + Alertmanager → Telegram webhook

- **Why:** per-metric thresholds, historical graphs, PromQL flexibility, evaluates every 15s.
- **How:** `apt install prometheus-node-exporter` (bind 127.0.0.1:9100), `apt install prometheus alertmanager`, scrape config + `alertmanager.yml` with Telegram receiver. Grafana Cloud free tier (50GB logs / 10k metrics) can host the dashboards without another box.
- **Cost:** zero cash, ~1h setup, adds ~60MB RSS on the VPS.
- **When to pick this:** after first real outage where we need >5-min resolution, or when we want separate alerts per metric rather than "something is over threshold".
- **Out of scope for #345** — tracked in `hetzner-all.md` Phase 7 as a future P7.3 upgrade.

**This runbook executes Option A.** Option B is stubbed at the end for future reference.

### Step 2 — Install and configure PM2 log rotation

```bash
ssh root@46.224.219.3 <<'SH'
  # Install module (idempotent — re-running is safe).
  pm2 install pm2-logrotate

  # Policy: 10 MB per file, gzip on rotate, retain 30 days, daily rotate at 00:00.
  pm2 set pm2-logrotate:max_size 10M
  pm2 set pm2-logrotate:retain 30
  pm2 set pm2-logrotate:compress true
  pm2 set pm2-logrotate:dateFormat YYYY-MM-DD_HH-mm-ss
  pm2 set pm2-logrotate:workerInterval 30         # scan every 30s for size-based rotate
  pm2 set pm2-logrotate:rotateInterval '0 0 * * *' # daily at midnight (server TZ)
  pm2 set pm2-logrotate:rotateModule true         # rotate pm2-logrotate's own log too

  # Verify
  pm2 conf pm2-logrotate
  ls -la /var/log/pm2/
SH
```

**Expected:** `pm2 conf` prints the seven settings above; `/var/log/pm2/janicka-shop-{out,error}.log` exist (the live files), and on first daily run `…-<date>.log.gz` files start appearing.

2. **Log rotation dry-run** — `logrotate -d /etc/logrotate.d/janicka-shop` on VPS returns zero errors; forced rotation produces `.1.gz` files within 30s; `nginx -t` still passes after the `kill -USR1`.
3. **PM2 size-trigger rotate** — artificially grow a PM2 log and verify rotation:
   ```bash
   ssh root@46.224.219.3 '
     yes "pm2 logrotate size-test" | head -c 11000000 >> /var/log/pm2/janicka-shop-out.log
     sleep 45        # pm2-logrotate workerInterval is 30s
     ls -la /var/log/pm2/ | head
   '
   ```
   Expect a new `janicka-shop-out__<timestamp>.log.gz` beside a truncated live file.
4. **CPU threshold drill** — run `stress-ng` to push 1-min load > 80% of cpu_count:
   ```bash
   ssh root@46.224.219.3 '
     apt-get install -y stress-ng                         # first time only
     stress-ng --cpu $(nproc) --timeout 600s &
     sleep 120
     curl -fsS -H "Authorization: Bearer $CRON_SECRET" \
       http://127.0.0.1:3000/api/health-extended | jq .breaches
   '
   # Expect breaches.cpu: true within 2 minutes.
   ```
   Within ≤ 15 min (5-min poll × 2 failures + slack), Telegram receives `*DOWN* — janicka-shop thresholds (origin)`. After stress-ng ends, next successful poll flips it back to Up and Telegram receives the recovery message.
5. **Disk threshold drill (DESTRUCTIVE — skip unless you want to)** — fill `/var/log` to 85%+ with `fallocate -l …G /var/log/fill.bin`, verify breach, `rm /var/log/fill.bin` after.
6. **Log-dir size stability** — after all drills, `df -h /var/log` and `du -sh /var/log/pm2 /var/log/janicka-shop /var/log/nginx`. Should not exceed: ~10 rotated PM2 files × compressed + 30 rotated app-cron files ≈ tens of MB, NOT GB.

## Rollback

- **Undo PM2 logrotate** — `pm2 uninstall pm2-logrotate` on VPS. Live logs continue unrotated (will grow unbounded — re-install within days, don't leave rollback in place).
- **Undo system logrotate** — `rm /etc/logrotate.d/janicka-shop`. Default distro `/etc/logrotate.conf` weekly sweep still handles `/var/log/nginx/*.log` via Debian's packaged logrotate.d/nginx entry — but our `janicka-shop.*.log` names may not match that glob, so expect slow growth. Short-term rollback only.
- **Disable threshold monitor** — Dashboard → Monitor → "Running" OFF. Telegram stays silent; up/down P7.1 monitors continue untouched.

## Gotchas

1. **`CRON_SECRET` must match on both ends** — if UptimeRobot's header falls out of sync with `/opt/janicka-shop/.env.production`, the monitor gets a 401 on every poll and goes Down for the wrong reason. Rotate secret → update monitor header in same change window.
2. **PM2 logrotate has a bug on very old `pm2-logrotate@2.6` releases** — the daily cron expression is ignored if `workerInterval` is less than `rotateInterval`. The settings above (30s worker, daily cron) are the supported combo on current `pm2-logrotate`. Verify with `pm2 install pm2-logrotate` which pins the latest.
3. **`copytruncate` vs `create`** — used `copytruncate` for `/var/log/janicka-shop/*.log` because the cron scripts append via `>> cron.log` (no SIGHUP handler) — a rename-then-create would leave the scripts writing into the rotated file. For nginx we use the `create` + `kill -USR1` dance because nginx handles SIGUSR1 to reopen log FDs.
4. **stress-ng on a tiny VPS can OOM the Node app** — if PM2 restarts the app during the drill, the test still proves the alert path works, but now also produces a spurious P7.1 origin-down alert. Run the CPU drill outside peak hours, and cap CPU with `--cpu 3` on a 4-vCPU box if you want to preserve Node's schedule.
5. **Disk threshold edge case on zfs/overlayfs** — `fs.statfs('/')` reports the overlay, not the underlying. If we later move /opt to a separate mount, point the route at that path explicitly. Current Hetzner image is a single ext4 root, so fine.
6. **Telegram alert body reuses P7.1 template** — `*monitorAlertType* — *monitorFriendlyName*` means a threshold breach shows up as `*DOWN* — janicka-shop thresholds (origin)`. Distinct friendly name is the only way to tell it apart from the health-endpoint down alert. Keep them distinct.
