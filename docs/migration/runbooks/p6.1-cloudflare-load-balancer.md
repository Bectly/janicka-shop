## Pre-flight checklist (DO NOT start LB setup until all pass)

1. **Zone active in Cloudflare** — `jvsatnik.cz` nameservers already delegated to CF (shared with `jarvis-janicka.jvsatnik.cz` CF Tunnel), zone status "Active".
2. **Hetzner origin healthy** — `curl -sS https://jvsatnik.cz/api/health` via CF-proxied record returns `{"status":"ok",...}` 200 (P3.4 smoke-test signed off).
3. **Vercel origin healthy** — `curl -sS https://janicka-shop.vercel.app/api/health` returns 200 with same JSON shape.
4. **Vercel domain binding removed / not conflicting** — `jvsatnik.cz` is NOT attached to the Vercel project (otherwise Vercel 404s non-Vercel hosts or CF origin lookup gets confused). Vercel project keeps only `*.vercel.app`.
5. **Cloudflare plan supports Load Balancing** — Free plan does NOT include LB. Either:
   - Enable LB add-on on Free plan (pay-as-you-go) at dash → Traffic → Load Balancing → Enable.
   - Or upgrade zone to Pro ($25/mo) which still requires LB add-on separately.
   - Confirm billing method on file before proceeding (LB bills per hostname + per query).
6. **CF API token (optional but recommended)** — scope: `Zone.Load Balancing:Edit` + `Zone.Health Checks:Edit` for this zone. Save to JARVIS DB as `cloudflare-janicka-lb`. Lets us script future changes via `cf-terraforming` or API.

## Execution

All steps via Cloudflare dashboard unless noted. Order matters — create health monitor BEFORE pool, pool BEFORE LB.

### Step 1 — Create health monitor

Dashboard → Traffic → Load Balancing → **Monitors** → Create.

| Field | Value |
|---|---|
| Name | `janicka-health` |
| Type | `HTTPS` |
| Method | `GET` |
| Path | `/api/health` |
| Port | `443` |
| Host header | `jvsatnik.cz` |
| Interval | `30s` |
| Timeout | `5s` |
| Retries | `2` (→ 2 consecutive failures before marking unhealthy = matches task acceptance) |
| Expected codes | `200` |
| Expected response body | `"status":"ok"` (substring match — covers JSON order drift) |
| Follow redirects | OFF |
| Regions to probe from | `WEU` (Western Europe) + `EEU` (Eastern Europe) — cheapest, covers CZ user path |

**Why these values:** `/api/health` already returns `Cache-Control: no-store` and the `status:"ok"` substring, so no code change needed. 30s interval × 2 retries = ~60s detection window, matches acceptance criterion ("within 60s traffic flows to Vercel"). Timeout 5s is comfortable headroom over typical Next.js cold-start of 200-800ms.

### Step 2 — Create Hetzner origin pool

Dashboard → Traffic → Load Balancing → **Pools** → Create.

| Field | Value |
|---|---|
| Name | `pool-hetzner` |
| Description | `Primary — kryxon VPS, Debian, PM2, nginx, Redis` |
| Health check regions | `WEU` + `EEU` (match monitor) |
| Minimum origins healthy | `1` |
| Notification email | `jkopecky666@gmail.com` |

Add origin inside the pool:

| Field | Value |
|---|---|
| Name | `hetzner-kryxon` |
| Address | `46.224.219.3` |
| Weight | `1` |
| Enabled | ✅ |
| Host header override | `jvsatnik.cz` |
| Monitor | `janicka-health` (from Step 1) |

**Why host header:** nginx vhost on kryxon is name-based (`server_name jvsatnik.cz`) — without the override, the probe hits the default server block and may 404 or hit the wrong project.

Dashboard → Traffic → Load Balancing → **Pools** → Create.

| Field | Value |

Add origin:

| Monitor | `janicka-health` |

### Step 4 — Create the Load Balancer

Dashboard → Traffic → Load Balancing → **Load Balancers** → Create.

| Field | Value |
|---|---|
| Hostname | `jvsatnik.cz` (apex) |
| Proxy (orange cloud) | ON |
| TTL | `Auto` (LB ignores TTL when proxied) |
| Default pools (ordered failover) | 1) `pool-hetzner`  2) `pool-vercel` |
| Fallback pool | `pool-vercel` |
| Session affinity | `ip_cookie`, TTL 1800s (matches cart reservation window; prevents mid-checkout bouncing across origins — per directive). `samesite=Lax`, `secure=Always`, `drain_duration=60s`, `zero_downtime_failover=sticky`. |
| Traffic steering | `Off / Failover` (NOT geo/dynamic — we only have one primary region) |
| Adaptive routing | OFF (single-region, adds noise) |

Repeat for `www.jvsatnik.cz` → separate LB with same pools (or CNAME `www` → `@` at the zone level after apex LB exists; both work, separate LB is cleaner for future geo routing).

**DNS record conflict warning:** when you create the LB on `jvsatnik.cz`, Cloudflare will prompt to delete the existing `A  @  46.224.219.3` record that P2.1 set up — accept. The LB now owns that hostname and serves the right origin IP dynamically. Do NOT re-add a plain A record while the LB exists.

## Acceptance tests

Run from a machine NOT on Cloudflare's network (home connection is fine).

### Test 1 — Both origins healthy

```bash
# LB dashboard should show: pool-hetzner HEALTHY (green), pool-vercel HEALTHY (green).
# Traffic should land on Hetzner (primary) — verify via commit hash:
curl -sS https://jvsatnik.cz/api/health | jq .
# Expect: "commit" ≠ VERCEL_GIT_COMMIT_SHA → Hetzner's GIT_COMMIT_SHA (set via env sync).
```

### Test 2 — Failover (Hetzner down)

```bash
ssh kryxon 'sudo systemctl stop nginx'

# Wait up to 60s (2 × 30s probe interval), then:
curl -sS https://jvsatnik.cz/api/health | jq .
# Expect: "commit" == VERCEL_GIT_COMMIT_SHA (Vercel serving).
# Dashboard: pool-hetzner UNHEALTHY (red), pool-vercel HEALTHY (green), alert email received.
```

### Test 3 — Recovery

```bash
ssh kryxon 'sudo systemctl start nginx'

# Wait up to 60s:
curl -sS https://jvsatnik.cz/api/health | jq .
# Expect: commit flips back to Hetzner's. Dashboard: pool-hetzner HEALTHY.
```

### Test 4 — No split-brain during recovery window

During the 60s detection window in Test 2, some requests may still land on Hetzner and time out. Verify the timeout is below CF's edge timeout (100s default) so users see the failover, not a naked 524.

```bash
# While Hetzner is stopped:
for i in {1..20}; do curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" https://jvsatnik.cz/; sleep 3; done
# Expect: mix of 200s — ZERO 524/520/522 after the ~60s mark.
```

## Rollback

If LB misbehaves (endless flapping, cost spike, wrong origin picked):

1. Dashboard → Load Balancing → disable the LB for `jvsatnik.cz` (toggle Enabled OFF).
2. Cloudflare auto-falls-back to a plain A record — **re-add** `A  @  46.224.219.3  Proxy: ON`.
3. Hetzner serves 100% of traffic directly; Vercel remains warm but unused.
4. Investigate, adjust monitor thresholds, re-enable.

Emergency full-bypass to Vercel (Hetzner completely dead + LB not failing over):
- Change DNS `A` to Vercel's anycast IPs (check `dig a janicka-shop.vercel.app +short`) OR CNAME `@` → `janicka-shop.vercel.app` (CF supports CNAME flattening).
- Remember to attach the domain to the Vercel project first (`vercel domains add jvsatnik.cz`) or Vercel 404s the host.

## Known gotchas

- **CF edge cache + health probe race:** if nginx returns `Cache-Control: public, max-age=X` on `/api/health`, CF edge might serve stale OK after origin dies. Our `/api/health` already sets `no-store, no-cache, must-revalidate` (see `src/app/api/health/route.ts`) — do not change this.
- **Vercel cold-start during failover:** first request hitting a dormant Vercel deployment can take 2-4s. CF LB probes are synthetic (keep it warm), so Vercel should stay warm as long as probes run. If Vercel scales to zero anyway, `Cache-Control: no-store` on health route prevents the probe from getting cached.
- **IPv6:** Hetzner VPS has IPv6 (`2a01:4f9:...`). Add as a second origin in `pool-hetzner` with the same host-header + monitor to get dual-stack failover within the pool. Optional — most CF edge traffic is IPv4 anyway.
- **Vercel auth protection:** if the Vercel project has Deployment Protection enabled, `/api/health` returns 401 → LB marks Vercel unhealthy → no failover target. Check `vercel.json` or project settings, disable protection for the production domain alias.
- **Host-header mismatch on Vercel origin:** if requests to Vercel arrive with `Host: jvsatnik.cz` but no domain binding, Vercel routes to a 404 project-not-found page. This is why Step 3 leaves the host override BLANK — CF passes through `janicka-shop.vercel.app`.
