# P7.1 — UptimeRobot monitors + Telegram alerts via JARVIS bot

**Task:** #344 — 5-minute HTTPS monitors on `/api/health` (via CF-proxied domain AND direct Hetzner IP), keyword check `"status":"ok"`, webhook alert to `@Jarvis_bectly_bot` on any down state.
**Why two monitors:** CF-proxied hit validates the full edge → nginx → PM2 → Next.js chain. Direct-IP hit bypasses Cloudflare — catches origin failures that CF is masking via cached HTML, Always Online, or LB failover to Vercel. Without the direct monitor, a Hetzner-only outage looks green to end users but we lose the one box we actually control.
**Status:** BLOCKED on (a) domain purchase `janicka-shop.cz` (#329), (b) P3.4 Hetzner smoke-test (#334) — only the CF-proxied monitor depends on the domain; the direct-IP monitor can be armed immediately once PM2 serves on :443 or via CF Origin Rule to expose `/api/health` on the bare IP over HTTPS. See "Interim single-monitor mode" below.

2. **Telegram bot reachable** — `@Jarvis_bectly_bot`, chat_id `8750673812`, token lives in JARVIS DB:
   ```bash
   sqlite3 ~/.claude/jarvis-gym/jarvis.db \
     "SELECT key_value FROM api_keys WHERE name='telegram-bot';"
   ```
   **NEVER** paste the token into the UptimeRobot webhook URL body field; put it in the URL path itself (UptimeRobot stores webhook URLs encrypted at rest). Verify the bot responds:
   ```bash
   TELEGRAM_BOT_TOKEN="$(sqlite3 ~/.claude/jarvis-gym/jarvis.db \
     "SELECT key_value FROM api_keys WHERE name='telegram-bot';")"
   curl -sS "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/sendMessage" \
     -d "chat_id=8750673812" \
     -d "text=🧪 UptimeRobot pre-flight test $(date -u +%FT%TZ)"
   # Expect: {"ok":true, ...}
   ```
3. **Health endpoint shape confirmed** — `GET /api/health` returns JSON containing literal string `"status":"ok"` (no spaces after colon in JSON.stringify default output). Verify once:
   ```bash
   curl -sS https://janicka-shop.cz/api/health | grep -o '"status":"ok"'
   curl -sS -k -H 'Host: janicka-shop.cz' https://46.224.219.3/api/health | grep -o '"status":"ok"'
   ```
   Both must print `"status":"ok"` for the keyword check to match. The `-k` skips TLS validation for the IP hit; UptimeRobot has its own "Ignore SSL errors" toggle for the direct-IP monitor.

## Execution

### Step 1 — Create Alert Contact (Telegram webhook)

Dashboard → My Settings → **Alert Contacts** → Add Alert Contact.

| Field | Value |
|---|---|
| Alert Contact Type | `Webhook` |
| Friendly Name | `JARVIS Telegram — bectly` |
| URL to Notify | `https://api.telegram.org/bot<TELEGRAM_BOT_TOKEN>/sendMessage` |
| POST Value (JSON Format: ON) | see payload below |
| Send as JSON | ✅ ON |
| Enable notifications for | ✅ Down, ✅ Up, ✅ SSL errors |

**POST body (JSON, with UptimeRobot variables):**
```json
{
  "chat_id": "8750673812",
  "parse_mode": "Markdown",
  "text": "*monitorAlertType* — *monitorFriendlyName*\n\nStatus: `alertDetails`\nURL: monitorURL\nTime: alertDateTime UTC\n\nMonitor ID: monitorID"
}
```

UptimeRobot substitutes `*monitorFriendlyName*`, `*monitorAlertType*` (down / up), `alertDetails`, `monitorURL`, `alertDateTime`, `monitorID` at send time. Markdown bold around friendly name + alert type makes the Telegram message scannable on a phone lockscreen.

**Test:** click "Test Alert Contact" → Telegram should receive a test message within ~10s. If Telegram returns `chat not found`, the bot hasn't been started in a chat with bectly — open Telegram, search `@Jarvis_bectly_bot`, tap Start, then re-test.

## Acceptance tests

1. **Happy path** — both monitors show "Up" (green) after ~10 minutes of pulling `/api/health`.
2. **Down alert fires** — temporarily edit the health route to return 503:
   ```ts
   // src/app/api/health/route.ts (TEMPORARY)
   return NextResponse.json({ status: "drill" }, { status: 503 });
   ```
   Deploy. Wait 10 min. Expect **both** monitors to flip to "Down" (keyword `"status":"ok"` no longer matches AND HTTP status is 503). Telegram should receive two messages: one per monitor. Revert the route, redeploy, confirm both flip back to "Up" and Telegram gets two "Up" messages.
3. **CF-only failure simulation** — add a CF Transform Rule "Custom response" for path `/api/health` that returns 500 (overrides origin). Wait 10 min. Expect **edge monitor Down, origin monitor Up**. Telegram shows this asymmetry. Delete the Transform Rule after the drill.
4. **SSL expiry reminder** — on next certbot renewal, confirm UptimeRobot's SSL reminder fires ≥7 days before expiry (independent check of certbot renew-hook health). No action needed during deploy; this validates itself passively.

## Rollback

- **Disable a monitor** — Dashboard → Monitor → toggle "Running" OFF. Keeps config, stops checks. Use this during planned maintenance windows to avoid alert noise.
- **Delete the Telegram alert contact** — Alert Contacts → Delete. All monitors lose their channel instantly (UptimeRobot falls back to account email only). Use only if the bot is compromised.
- **Disable entirely** — pause both monitors, remove alert contact. Free tier has no cleanup cost.

## Interim single-monitor mode (pre-domain)

Until `janicka-shop.cz` lands (#329), we can still arm the direct-IP monitor today:

1. Expose `/api/health` on `https://46.224.219.3/api/health` — requires nginx listening on 443 with a temporary self-signed cert (Let's Encrypt can't issue for IPs). Snippet:
   ```nginx
   server {
     listen 443 ssl default_server;
     ssl_certificate     /etc/ssl/self-signed/hetzner.crt;
     ssl_certificate_key /etc/ssl/self-signed/hetzner.key;
     location = /api/health {
       proxy_pass http://127.0.0.1:3000;
       proxy_set_header Host janicka-shop.cz;  # satisfies Next.js Host check
     }
   }
   ```
2. UptimeRobot monitor with "Ignore SSL Errors: ON" passes despite untrusted cert.
3. When P2.2 certbot lands, swap to real cert, delete self-signed files, keep the monitor running.

This unblocks early warning on Hetzner-side outages during Phase 3/4/5 buildup without waiting on domain purchase.

## Gotchas

1. **Chat ID is user-specific** — `8750673812` is bectly. If we ever want Janička to receive alerts too, either add a second webhook alert contact with her chat_id OR create a Telegram group, add the bot + both users, and use the group's chat_id (starts with `-100...`).
2. **Telegram rate limit** — bot API allows ~30 messages/sec globally, 1 msg/sec to same chat. Two monitors at 5-min interval = zero risk, but a flapping monitor could spam. UptimeRobot's "alert after N failures" dedupes this.
3. **Keyword check is substring match** — `"status":"ok"` matches `{"status":"ok",...}`. If we ever return `{"status": "ok"}` (with a space after the colon — e.g., if someone switches to pretty-printed JSON), the keyword stops matching and both monitors go red with ZERO actual outage. Anchor test: keep `src/app/api/health/route.ts` using `NextResponse.json(…)` default serialization (no `JSON.stringify(…, null, 2)`).
4. **Free-tier 5-min interval** — worst-case 10-minute detection (2 failures × 5 min). Acceptable for an early-stage shop. If we later need 1-min detection (Pro at $7/mo) or 30s (LB already does 30s with Vercel failover in P6.1), upgrade the tier — do NOT try to work around with a second free account (UptimeRobot ToS forbids it).
5. **Public status page** (optional, free) — UptimeRobot offers a public status page at `stats.uptimerobot.com/<ID>`. For a consumer-facing shop this builds trust ("Janička je transparentní i o výpadcích"). Can be added post-launch; enable via Dashboard → Status Pages. Skip during pre-launch to avoid showing early instability.
6. **Domain-less alert noise** — if we arm the edge monitor before DNS points at CF, it will show "Down" indefinitely and spam Telegram on creation. DO NOT create the edge monitor until post-#329 DNS cutover. Direct-IP monitor is safe to create now (once interim nginx 443 is up).
