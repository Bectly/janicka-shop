## Policy change: Failover ‚Üí Random (weighted)

P6.1 configured the LB with **Traffic Steering: Off / Failover**. Weighted split requires **Random** steering with weights applied per pool.

Dashboard ‚Üí Traffic ‚Üí Load Balancing ‚Üí edit `jvsatnik.cz` LB:

| Field | Before (P6.1) | After (P6.3) |
|---|---|---|
| Traffic steering | `Off / Failover` | `Random` |
| Default pools | `[hetzner-primary, vercel-standby]` ordered | both listed, weights applied |
| Fallback pool | `vercel-standby` | `vercel-standby` (unchanged) |
| Session affinity | `ip_cookie` (TTL 1800s) | `ip_cookie` (TTL 1800s) ‚Äî unchanged |

Weights are set **per-pool-reference inside the LB**, not inside the pool itself. When you switch to Random, CF reveals a "Weight" slider on each pool row. Weights are proportional, not absolute percentages ‚Äî `hetzner=1 vercel=9` means 10/90 split.

**Important:** fallback pool still kicks in if BOTH weighted pools go unhealthy. Leave it as `vercel-standby` ‚Äî the health monitor guarantees an unhealthy pool gets 0% regardless of weight.

### Verify the split

From two different networks (home + mobile hotspot), over 5 minutes:

```bash

  curl -sS https://jvsatnik.cz/api/health \
    | jq -r '.commit // "null"'
  sleep 2
done | sort | uniq -c
```

Expect roughly 5 Hetzner / 45 Vercel (10/90). Small-sample noise is fine; anything outside 3-8 Hetzner suggests routing broke ‚Äî halt and rollback.

### Monitor for 24h

Check every 4h (or set Telegram alerts ‚Äî see P7 when it ships):

| Signal | Source | Threshold (abort if exceeded) |
|---|---|---|
| 5xx rate | CF dashboard ‚Üí Analytics ‚Üí Status codes, filter last hour | **>2√ó Vercel baseline** for 2 consecutive 10-min windows |
| p95 TTFB | CF Analytics ‚Üí Performance | **>1.5√ó Vercel baseline** sustained 30min |
| Sentry new issues | Sentry ‚Üí janicka-shop ‚Üí Issues ‚Üí "First seen: last 24h" | **any new unique issue** with >5 events ‚Äî investigate before stage 2 |
| PM2 memory | `ssh kryxon 'pm2 status'` | RSS stable, no restart loop. If `restarts` climbs >3 in 24h, abort. |
| DB query count | Turso dashboard ‚Üí `janicka-shop` DB ‚Üí Metrics | should scale ~linearly with Hetzner share (~10% extra queries vs Vercel-only). Spike >30% uplift = N+1 regression, abort. |

Revert pool weights to `hetzner-primary=0 vercel-standby=1` (or flip steering back to Failover ‚Äî functionally equivalent when Hetzner weight is 0). Hetzner stays warm, investigate from PM2 logs / Sentry at leisure.

## Known gotchas

- **CF edge cache skew between origins:** if any page is cached at the CF edge (set `Cache-Control: public, max-age=X` from origin), users hit the edge, not the origin. During weighted split, two consecutive requests from the same visitor might not route to different origins because the first one filled the cache. Current state: only `/api/health` is `no-store` ‚Äî most pages are ISR-rendered but served via CF at edge-cache defaults. Impact: real-user latency mostly reflects CF edge, not origin. **Mitigation:** rely on `/api/health` and admin routes (never cached) for routing verification, not on public pages.
- **Session affinity ‚Äî `ip_cookie` TTL 1800s:** C4378 flipped this ON to avoid bouncing a user mid-checkout between origins during weighted splits. Cart is client-side (Zustand ‚Üí localStorage) and reservations are server-side (Prisma / Turso) ‚Äî affinity isn't required for correctness, but it keeps CF Analytics per-origin numbers interpretable and prevents double-charge edge cases at a payment callback. Stays on through all three stages.
- **Vercel staying warm:** as long as CF LB probes keep pinging Vercel's `/api/health` every 30s, Vercel should not cold-start. If Vercel scales to zero despite probes, add a scheduled UptimeRobot ping as insurance (P7.1 #340).
- **WebSocket upgrade mid-shift:** admin panel has no WS yet, but `devChat` uses long-polling and webhook inserts. Switching origins between polls is safe because the DB is the source of truth (Turso). No sticky needed.
- **Turso rate limits:** free tier is 500M rows read / 10M rows written per month. At 100% Hetzner, we lose the Vercel ‚Üí Turso read-replica traffic pattern and gain the Hetzner VPS direct connection. Prisma's query count should stay roughly the same, but watch the Turso dashboard billing page on day 3.
- **DNS propagation myth:** CF LB does not have meaningful DNS propagation ‚Äî weight changes apply at the CF edge within seconds. The ‚â§30s TTL is mostly cosmetic. If after 2 min you still see 100% old behavior, check that you saved the LB config, not just the pool config.
