## 3. Static-asset best practices

| Asset class | Cache-Control | Encoding | Verdict |
|---|---|---|---|
| HTML (`/`) | `s-maxage=60, swr=300` | **brotli** ✅ | OK |
| `_next/static/chunks/*.js` | `max-age=31536000, immutable` ✅ | **gzip only** ⚠ | gzip works but brotli would save ~15-20% on text bundles |
| `_next/static/chunks/*.css` | `max-age=31536000, immutable` ✅ | **gzip only** ⚠ | same as JS |
| `/uploads/products/*.webp` | `max-age=31536000, immutable` ✅ | n/a (binary) | OK |

**Origin nginx (1.18.0)** has only `gzip on;` — no brotli module compiled in (`/usr/lib/nginx/modules/` lists 6 modules, none of which are `ngx_brotli`). HTML responses come back as `content-encoding: br` because Cloudflare re-compresses at the edge; static `_next/*` assets are served by the Next.js process (not nginx static handler) and stay gzip end-to-end.

**Other best practices verified**:
- HTTP/2 ✅ (origin `listen 443 ssl http2`, CF speaks h2 to client + advertises h3 via `alt-svc`).
- HTTP/3 NOT enabled at origin (nginx 1.18 cannot do quic). CF advertises h3 to clients but the back-channel is h2 — fine.
- HSTS `max-age=63072000; includeSubDomains` ✅ (2y).
- `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy` ✅.
- CSP set, includes Comgate/Packeta/GA/Pinterest/Facebook ✅.
- Critical CSS via beasties (issue #517) ✅ — confirmed in DOM (`<style>` blocks inline in head).
- `manifest.webmanifest`, `robots.txt`, `sitemap.xml` all 200 ✅.
- TLSv1.3, ECDSA cert, OCSP fine ✅.

## 4. Top problems sorted by impact

### P0 — LCP blocker on mobile (both home + /products)

**Symptom**: Mobile LCP 7.2-7.9 s. Desktop LCP fine (1.5-2.3 s).

**Root cause**: The product card images on home + /products grid are served as raw `/uploads/*.webp` files at 130-200 kB each, NOT through `/_next/image`. The grid loads ~8 above-the-fold images on mobile = **~1.3 MB of images before LCP**. Total mobile transfer for /products = **4,125 kB** (large for a 4G profile).

**Network audit (mobile /products)**:
- 8 product .webp files in top transfers, 130-198 kB each, all served raw at fixed dimensions.
- Total page weight: 4.1 MB — Lighthouse "Avoid enormous network payloads" fires.
- 33 scripts on first load.

**Recommended fix** (BOLT follow-up, NOT done in this audit):
- Wrap `<ProductCard>` images in Next/Image and let `/_next/image` resize them per breakpoint with `sizes="(max-width: 768px) 50vw, 25vw"`. Confirmed in C5171 audit that `unoptimized={true}` is currently set on cards/PDP/lightbox. That guarantee was conscious (preserve quality) but is the direct LCP killer on mobile.
- Alternatively: pre-generate 400/600/800 width WebP variants on upload and serve via responsive srcset. R2 doesn't have an image-resize CDN, so Next/Image is the lighter path.

### P1 — React hydration errors on prod

Lighthouse console errors on every load (verified on home + /products):
- **Home mobile (3×)**: `Minified React error #419` — a Suspense boundary errored during SSR, React fell back to client-rendering the root.
- **/products mobile (2×)**: `Minified React error #418` — hydration mismatch (server HTML ≠ client tree).

These cost runtime perf (extra client renders) and hint at SSR/CSR data drift. Likely candidates: cookie-dependent banners, GA/Pinterest tags, cart count, time-elapsed badges, or any `Date.now()` in RSC. Worth a Bolt task to grep `useEffect` + suspense boundaries in homepage + grid components.

### P2 — Unused JavaScript (~109 kB)

Two chunks dominate:
- `/_next/static/chunks/0xc1al8zsp-_y.js` — 124 kB transferred, **87 kB wasted (70%)**.
- `/_next/static/chunks/0956.uguclzfz.js` — 71 kB transferred, **24 kB wasted (34%)**.

This is bundle bloat — 70% unused on a 124 kB chunk means a heavy lib (probably analytics, framer, or admin code) is being shipped to public pages. Recommend dynamic-import or move to admin route group.

### P3 — Render-blocking CSS (612 ms / 166 ms)

Two CSS files on the critical path: `0hoxfe45wvwxu.css` (612 ms wasted on home mobile), `16tqqewvcwt_v.css` (162 ms). Beasties already inlines critical CSS, so the rest can be deferred via `<link rel="preload" as="style"` or split. Worth investigating.

### P4 — Logo aspect-ratio mismatch (a11y/BP)

`/_next/image?url=%2Flogo%2Flogo-header.png&w=256&q=75` displayed at 120×48 but actual asset is 88×48. Image is being stretched horizontally. Easy fix: align `width`/`height` props in `<Image>` with the asset's true aspect ratio.

### P5 — Color contrast & a11y nits

- Home mobile: 20 color-contrast violations.
- /products mobile: 7 color-contrast violations + heading-order out of sequence.
- Both pages: 1 button without an accessible name.

Not blocking but Lighthouse a11y stuck at 91-92, would push to 95+ with these.

### P6 — Resource hints

No `dns-prefetch`/`preconnect` to third-party domains observed. Current `<link rel="preload">` covers logo + 1 low-priority chunk. Adding preconnect to widely-used 3rd parties would shave ~50-100 ms on first paint:
- `https://www.googletagmanager.com`
- `https://connect.facebook.net`
- `https://s.pinimg.com`
- `https://payments.comgate.cz` (only on /checkout)
- `https://widget.packeta.com` (only on /checkout)

(Don't add to public pages where they're not used — checkout-only.)

### P7 — nginx observability gap

`log_format` does not log `$upstream_cache_status`. Without it, "what's our hit ratio?" can't be answered without rebuilding nginx + brotli or running a full proxy log to ELK. Trivial fix: add a custom log_format that appends `cs=$upstream_cache_status` and reload nginx. No deploy risk.
