**Thresholds (Google 2024+ field)**: LCP ≤ 2.5 s · INP ≤ 200 ms · CLS ≤ 0.1. ⚠ = needs-improvement, ❌ = poor.

> **Notes**
> - **INP is a field-only metric.** Lighthouse lab reports **TBT** (Total Blocking Time) as a proxy. Map: TBT ≤ 200 ms ≈ INP good; TBT > 600 ms reliably predicts INP poor.
> - **Cart state.** `/kosik/` hydrates from Zustand `localStorage` — a cold Lighthouse run measures the *empty-cart* view. Real cart-with-items will be equal or worse (item cards + R2 images + sticky summary layout).
> - **Route URL drift.** `/produkty` is a 308 redirect to `/products`; same for `/kosik` → `/cart`. Internal app links already use the English canonical (51 × `/products|/cart` vs 2 × Czech in home HTML), so the redirect cost is paid mainly by: (a) external / SEO entry, (b) bookmarked URLs, (c) the `/produkty/` → `/produkty` → `/products` double hop added by Vercel trailing-slash normalisation. Lighthouse attributes ~765 ms of mobile LCP on every redirected route to this chain.

INP proxy (TBT) is within budget everywhere — JavaScript execution is not the primary bottleneck. **The problems are render-path / layout / image priority issues, not bundle size.**

### Shared findings across all mobile routes

- **Render-blocking CSS**: `_next/static/chunks/0r26w~o~8fc-6.css` (32 KB) blocks first paint for ~300 ms on every route. This is the Tailwind-compiled bundle.
- **Legacy JS polyfills**: `_next/static/chunks/0956.uguclzfz.js` ships ~14 KB of legacy transforms (`Array.prototype.at` polyfill among others) even on modern browsers. Upgrade browserslist target or `swcMinify` modern output.
- **Unused JS**: ~46 KB (home), ~65 KB (listing/pdp/cart/checkout) — consistent across routes, so it's framework/layout-shell code not gated by route.
- **No `preconnect` hint** to the R2 image CDN (`pub-88d95c0ca85d4cb999122434d83fb3c9.r2.dev`) despite it being on every page.
- **Cookie-consent `<p>` is the LCP element on /, /produkty/, /kosik/, /checkout.** Because the banner is a client component that paints post-hydration, it becomes the largest paint late and inflates LCP even on routes where it's objectively not the hero content. `/produkty/[slug]` is the only route where the LCP is a real product image.

### #1 — Homepage CLS 0.431 ❌ (4.3× budget)

**Diagnosis.** Lighthouse identifies 4 layout shifts in the `div.animate-page-enter` subtree of `src/app/(shop)/page.tsx`:

| Shift score | Element |
|---|---|
| 0.278 | `main#main-content > div.animate-page-enter > section.mx-auto` → *Kategorie* section |
| 0.153 | `main#main-content > div.animate-page-enter > div` → *Naše sliby* section |
| 0.000 + 0.045 | `main#main-content > div.animate-page-enter > div` → *Populární značky* (H&M, adidas, Zara…) section, `<main>` itself |

All four shifts happen because the `animate-page-enter` wrapper plays an entry animation (opacity/translate) while its section children reveal *after* font/image load, re-flowing the document. Mobile users see the hero, then everything below jumps ~450–835 px as brand tiles and category cards resolve.

This is by far the single worst quality-of-experience regression on prod: 0.431 is in Google's "poor" band (> 0.25).

**Draft directive.**

> **[PERF-HOME-CLS] BOLT — Homepage CLS 0.431 → ≤0.1 on mobile**
>
> Root cause: `animate-page-enter` entry animation in `src/app/(shop)/page.tsx` runs while below-the-fold sections (Kategorie, Populární značky, Naše sliby) resolve their final height → 4 large shifts totalling 0.431 CLS.
>
> Scope (no API/DB changes):
> 1. Audit every direct child of `<main className="...animate-page-enter">` on homepage — the brand grid, category grid, "Naše sliby" grid. Add a `min-h-[Xpx]` skeleton (matching the rendered height in Chrome devtools @ 375px) OR wrap image/count data in a suspense boundary that reserves space.
> 2. Verify brand tile images (`<img>` children of the "Populární značky" grid) have explicit `width`/`height` attributes so reserved space doesn't collapse after fetch.
> 3. Disable `animate-page-enter` entry translate/fade OR run it via `transform` only (CSS transform does not trigger layout shift).
>
> Accept: rerun `npx lighthouse https://janicka-shop.vercel.app/ --quiet` → mobile CLS ≤ 0.1, layout-shifts audit = 0 shifts or score ≤ 0.045. Mobile Perf ≥ 85.

### #2 — Product-detail LCP 5.04 s ❌ (2× budget)

**Diagnosis.** LCP element on `/produkty/[slug]` *is* the hero product image (`page-0-IMG`, `<img alt="Pánská zimní bunda CXS (vel. XS) — fotka 1">`). LCP sub-parts from Lighthouse `lcp-breakdown-insight`:

| Sub-part | Duration |
|---|---|
| TTFB | 100 ms ✅ |
| **Resource load delay** | **2710 ms ❌** |
| Resource load duration | 42 ms ✅ |
| Element render delay | ~190 ms ✅ |

The image itself transfers in **42 ms** once started. The ~2.7 s of "load delay" is the browser not knowing the image is critical — it only discovers and schedules the request after HTML + CSS + main JS chunks. Root cause is almost certainly that the Next.js `<Image>` in the PDP gallery does not have `priority` set on the first slide, so Next emits `loading="lazy"` / `fetchpriority="auto"`. Lighthouse HTML inspection confirms the LCP `<img>` has `decoding="async"` with no `fetchpriority=high`.

An additional ~765 ms of the 5.04 s figure is the `/produkty/...` → `/products/...` 308 redirect.

**Draft directive.**

> **[PERF-PDP-LCP] BOLT — PDP mobile LCP 5.04 s → ≤2.5 s**
>
> Two compounding root causes:
> 1. First product-gallery image (`src/components/shop/product-gallery.tsx` or wherever `<Image>` renders slide 0) is not priority-hinted. LCP `resourceLoadDelay` is 2710 ms on mobile = image not discovered until after JS chunks. Fix: on the first slide only, set `priority` on the Next `<Image>` (this emits `fetchpriority="high"` + preload link). Do NOT set priority on subsequent slides.
> 2. Czech slug route `/produkty/[slug]` 308-redirects to canonical `/products/[slug]`, costing ~765 ms on mobile. Either remove the Czech alias (migrate any inbound SEO/sitemap/email links first) or short-circuit the redirect by serving the same page component under both paths.
>
> Also cheap wins bundled:
> - Add `<link rel="preconnect" href="https://pub-88d95c0ca85d4cb999122434d83fb3c9.r2.dev" crossOrigin="anonymous">` in `src/app/layout.tsx` `<head>` — currently Lighthouse flags zero preconnected origins.
>
> Accept: mobile LCP on `/produkty/panska-zimni-bunda-cxs-vel-xs` ≤ 2.5 s, `resourceLoadDelay` < 500 ms. Rerun with Lighthouse.

### #3 — Cookie-consent banner becomes LCP on 4 of 5 routes (mobile LCP 3.70–3.77 s ⚠)

**Diagnosis.** On `/`, `/produkty/`, `/kosik/`, `/checkout`, Lighthouse reports the LCP element as:

```
<p class="mt-2 text-sm leading-relaxed text-muted-foreground">
  Používáme cookies pro základní fungování webu. Analytické a marketingové cookies…
```

located in `body > div.bottom-above-nav > div.mx-auto > p.mt-2`. This banner is a **client component** rendered post-hydration — it paints ~3.7 s into the load on mobile and, being a large multi-line paragraph in the viewport, becomes the largest paint. On pages where the *real* hero should be LCP (homepage headline, listing filter/title block, cart empty-state or checkout stepper), the cookie banner is instead ranking as the largest contentful paint.

This isn't necessarily broken CLS — it's a misaligned LCP that inflates every route where the actual content is smaller than the banner paragraph.

**Draft directive.**

> **[PERF-COOKIE-LCP] BOLT — cookie banner dominating LCP on 4/5 mobile routes**
>
> The cookie-consent `<p>` (`body > div.bottom-above-nav > ... > p.mt-2`) is a client component that paints post-hydration and becomes the LCP element on `/`, `/produkty/`, `/kosik/`, `/checkout` (3.70–3.77 s mobile). It is bigger than the real hero on those routes.
>
> Two fix options (pick based on GDPR review of current copy):
> - **A. SSR the banner shell.** Render the banner markup server-side so it paints in the first HTML, with client JS only wiring the accept/reject handler. Eliminates LCP displacement entirely.
> - **B. De-prioritise the banner text.** Split the long paragraph into a heading (`<h2>`) + collapsed details button — shrinks the banner's visual area below other hero elements so Lighthouse picks the real content as LCP.
>
> Cheap bundled win (unblocks ~300 ms on every mobile route):
> - `_next/static/chunks/0r26w~o~8fc-6.css` (32 KB) is render-blocking for 300 ms on all 5 routes. Verify the critical CSS extraction in Next 16.2.3 is enabled (`experimental.optimizeCss` or equivalent) — with Tailwind v4 this usually requires no config but the bundle is still blocking render.
>
> Accept: mobile LCP on `/`, `/produkty/`, `/kosik/`, `/checkout` drops below 2.5 s. LCP element on those routes is no longer the cookie banner `<p>`.

### C4867 NEW-2 resolution (2026-04-24, task #484 close-out)

Mini-spike per C4866 recommendation: `curl -sL <pdp-url>` against one **live** PDP (`/products/letni-boty-na-klinku-graceland-vel-37`) and the **sold** PDP re-sampled in C4866 (`/products/panska-zimni-bunda-cxs-vel-xs`), then scan the SSR `<head>` for image preloads and `fetchPriority`.

**Result: identical output.** Both HTML responses ship exactly 2 preloads in `<head>` — `logo-header.png` (image) and `17fy4hsc~jqgq.js` (low-priority script). **Zero `<link rel="preload" as="image">` for the hero product image on either URL. Zero `fetchPriority="high"` on any element in `<head>`.** The `<main id="main-content">` tag contains only a `<template id="B:4">` Suspense placeholder; the gallery markup streams in a later chunk.

**Conclusion.** Hypothesis (a) from NEW-2 (sold-overlay `div.opacity-60` delaying discovery) is **ruled out** — the sold page is not structurally different from the live page at the SSR level. Hypothesis (b) is **confirmed**: the Next.js `<Image>` `priority` prop in `src/components/shop/product-gallery.tsx` does NOT hoist a head-preload because the component is rendered inside a Suspense boundary that streams after `</head>` has already flushed. The `fetchPriority="high"` attribute on the `<img>` itself (which C4866 verified) helps only marginally — by the time the streamed gallery arrives, the browser has already queued all the scripts ahead of it.

**Fix direction (for a future Bolt directive, out of scope for this Trace close-out):** either (1) move the PDP's first gallery image out of the streaming Suspense boundary so Next can emit the head preload at SSR time, or (2) manually declare the hero image preload via PDP `generateMetadata` → `metadata.other` (Next 16 supports `rel="preload"` via `other` with an `as="image"` entry), or (3) inline a custom `<link rel="preload" as="image" fetchPriority="high">` in the server-rendered PDP page above the Suspense boundary, resolved from the same data fetch that feeds the gallery (first image URL, R2 CDN). Estimated ~15 LoC for option 3, zero new deps.
