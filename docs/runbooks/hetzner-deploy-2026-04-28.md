## Env

`.env.production` (81 keys) was generated by `scripts/sync-env-to-hetzner.sh --confirm` from JARVIS DB + local `.env.local`. Three keys (`R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`) were empty in `.env.local`; pulled from `vercel env pull --environment=production` and sed-patched into the VPS file. Suggested follow-up: backfill these into the JARVIS DB / local `.env.local` so the next sync run is idempotent without manual patches.

`NEXT_PUBLIC_SITE_URL=https://janicka-shop.cz` — required to pass the `checkSiteUrlEnv` boot guard (`*.vercel.app` is rejected in production).

## Known caveats (Phase 1 only)

- **Auth**: NextAuth logs `UntrustedHost` for IP-based access because `NEXTAUTH_URL` is the canonical domain. Login flows will work once DNS points at the VPS — no app changes needed.
- **TLS**: clients hitting `https://46.224.219.3` see a `geo.kryxon.cz` cert (name mismatch). Use `-k` for testing. Real cert lands in Phase 2 after DNS.
- **Hetzner Redis**: still bound to `127.0.0.1:6379` and password-protected. Vercel does NOT use this Redis (would need a tunnel). The Hetzner app instance does — `db:ok, redis:ok` confirmed.
- **Pre-existing rsync**: a prior worker had rsync'd an x86_64 standalone bundle into `/opt/janicka-shop`. That tree was deleted before the fresh clone (mismatched native binaries for libsql/sharp on aarch64).

## Repro / rollback

Rebuild on VPS:
```bash
ssh kryxon
cd /opt/janicka-shop
git pull
npm ci --no-audit --no-fund
NODE_ENV=production npm run build
cp -r public .next/standalone/
cp -r .next/static .next/standalone/.next/
cp .env.production .next/standalone/.env.production
pm2 restart janicka-shop
```

Rollback (route traffic back to Vercel — this isn't live anyway):
```bash
ssh kryxon "pm2 stop janicka-shop"
# Vercel is still production; stopping the VPS app is harmless.
```
