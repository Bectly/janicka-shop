Audited the same 3 features from Bolt Cycle #2290, after the C2291 fix was applied:
1. /objednavka order lookup page
2. sendAdminNewOrderEmail() admin notification
3. NotifyMeForm + notify-actions.ts

## Fixed Bug 1 — MEDIUM (rate-limit key collision)

**Files**:
- `src/app/(shop)/objednavka/actions.ts` line 31
- `src/app/(shop)/order/lookup/actions.ts` line 24

**Bug**: Both `/objednavka` and `/order/lookup` used the same rate-limit key `order-lookup:${ip}`, but with different window sizes (60,000ms vs 300,000ms). Shared key means lookups on one route exhaust the limit for the other route. Users could be falsely rate-limited, or attackers could bypass the longer 5-min window by using the shorter 1-min window to refresh the bucket.

**Fix**: Changed `/objednavka` to use key `objednavka-lookup:${ip}` and `/order/lookup` to use `order-lookup-legacy:${ip}`.

## Fixed Bug 2 — LOW (wrong order number placeholder in /order/lookup form)

**File**: `src/app/(shop)/order/lookup/lookup-form.tsx` line 32

**Bug**: Placeholder showed `JAN-240401-ABCD` but the actual order number format generated by `generateOrderNumber()` is `JN-YYMMDD-XXXXXXXX` (e.g., `JN-260404-XXXXXXXX`). The placeholder would confuse users entering their order number.

**Fix**: Changed placeholder to `např. JN-260404-XXXXXXXX`.

## Fixed Bug 3 — LOW (missing toUpperCase() transform in /order/lookup)

**File**: `src/app/(shop)/order/lookup/actions.ts` line 8-9

**Bug**: The older `/order/lookup/actions.ts` used `z.string().trim().min(1, ...)` without `.toUpperCase()`. Since order numbers are stored uppercase in the DB (`JN-...`), a user entering lowercase would get "not found" even with the correct order number. The new `/objednavka/actions.ts` correctly applies `.toUpperCase()`.

**Fix**: Added `.max(30).transform((v) => v.trim().toUpperCase())` to the orderNumber field in `/order/lookup/actions.ts`.

## Observations — no bugs, no fixes needed

- `sendAdminNewOrderEmail()`: fire-and-forget pattern with `.catch()` correct. `ADMIN_NOTIFICATION_EMAIL` env var properly guarded. `paymentMethod: data.paymentMethod` correctly passes "cod" for COD orders since `isCod = data.paymentMethod === "cod"`.
- `NotifyMeForm`: dedup logic correct. `sizes` passed as JSON string from `product.sizes` — consistent with DB storage.
- `ProductNotifyRequest` schema and DB table are in sync (confirmed via sqlite3 query).
- `/objednavka/actions.ts` email comparison: `order.customer.email.toLowerCase() !== email` is correct — `email` is already `.toLowerCase()` from zod transform, and the other side is also lowercased.

## Why: How to apply

When two routes implement the same feature (lookup, search, etc.), always audit for rate-limit key collisions. New routes must use distinct keys from pre-existing routes, especially when window sizes differ. Always apply `.toUpperCase()` when matching stored-uppercase identifiers (order numbers, codes).
